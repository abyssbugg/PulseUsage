# v0.6.28 Release Readiness

Status: pending. Do not publish, tag, or bump the version from `0.6.27` until blockers below are cleared.

## Scope

This readiness pass adds automated provider metadata validation only.

No provider was added. No provider parser was changed. No real account payloads, local paths, tokens, emails, IDs, or raw request/response bodies are included here.

## New release gates

- `bun run test:provider-validator`
  - Fast offline tests for `scripts/validate-provider-metadata.mjs`.
- `bun run validate:providers`
  - Validates all real bundled provider manifests.
  - Skips `plugins/mock`.
  - Runs in CI after `bun run bundle:plugins` and before build/test.

## Validator behavior

Default mode is migration-safe. It fails hard schema/file/docs issues and warns when classification metadata is missing.

Checked for each real provider:

- `plugin.json` parses as an object.
- `schemaVersion` is `1`.
- `id` is kebab-case and matches the provider directory.
- `name` is non-empty.
- `version` is semver-like.
- `entry` is a safe relative path and exists.
- `icon` is a safe relative SVG path and exists.
- `brandColor`, when present, is a six-digit hex color.
- `lines` is non-empty.
- line `type`, `label`, and `scope` are valid.
- duplicate line labels are rejected.
- `primaryOrder` is only accepted on progress lines.
- duplicate progress `primaryOrder` values are rejected.
- links, when present, use non-empty labels and `http` or `https` URLs.
- `docs/providers/<id>.md` exists.
- metric classification metadata is validated when present.
- docs classification drift is rejected when manifest classification data exists.
- required classified metrics must be referenced by `plugin.test.js`.

## Strict mode migration path

Strict mode is available but not ready for CI yet.

Use either command during the metadata migration:

- `bun scripts/validate-provider-metadata.mjs --strict`
- `PULSEUSAGE_PROVIDER_METADATA_STRICT=1 bun run validate:providers`

Before enabling strict mode in CI:

1. Add classification metadata to every real provider manifest line.
2. Use only `Required`, `Optional`, or `Plan-dependent`.
3. Add concise evidence for each classified line.
4. Keep each provider doc `Metric classification` table in sync with manifest labels and classifications.
5. Add or update `plugin.test.js` fixture coverage for every `Required` metric.
6. Run strict validation and fix all errors.

Do not guess classifications. If evidence is missing, leave the provider in migration warning state and report the blocker.
